# Payment Enhancement & Finetuning Plan

## Problem Statement

The HP Tourism Homestay Portal uses a **browser-redirect** payment architecture with Himkosh (IFMIS e-Challan). Our server never communicates directly with Himkosh — the user's browser carries encrypted payment data to Himkosh and brings back the response via redirect.

This creates a critical vulnerability: if the browser redirect fails after payment (user closes tab, network timeout, server restart during callback), the payment is **successfully recorded on Himkosh** but **stuck as "initiated" in our system**. The applicant sees a pending/draft application despite having paid.

## Current Payment Architecture

```
┌─────────────┐     ① Encrypted form      ┌──────────────────┐
│  Our Server  │ ──────────────────────────▶ │  User's Browser  │
│  (Node.js)   │                            │                  │
│              │     ④ Browser redirect     │                  │
│              │ ◀────────── (UNRELIABLE) ── │                  │
└─────────────┘                             └────────┬─────────┘
                                                     │
                                              ② Form POST
                                              ③ Payment done
                                                     │
                                                     ▼
                                            ┌──────────────────┐
                                            │   Himkosh IFMIS  │
                                            │  e-Challan GW    │
                                            └──────────────────┘

Problem: Step ④ is browser-mediated and UNRELIABLE.
         If it fails, our system never learns about the payment.
```

### Why Payments Get Stuck

| Failure Scenario | Frequency | Preventable? |
|-----------------|-----------|-------------|
| User closes browser after paying on Himkosh | Common | ✅ Yes — with reconciliation |
| Network timeout during redirect back | Occasional | ✅ Yes — with reconciliation |
| Server restart during callback (PM2/hotfix) | During deploys | ✅ Yes — with reconciliation |
| Himkosh gateway slow/unresponsive | Rare | ❌ No — Himkosh issue |

### Payment Statuses (Normal vs Problem)

| Status | Normal? | Action Needed |
|--------|---------|--------------|
| `success` | ✅ Yes | None |
| `failed` | ✅ Yes | User can retry |
| `cancelled_by_applicant` | ✅ Yes | User chose to cancel |
| `initiated` (< 15 min old) | ✅ Yes | Payment in progress |
| **`initiated` (> 15 min old, paid on Himkosh)** | **❌ BUG** | **Needs reconciliation** |

## Proposed Solution: 3-Layer Reconciliation

### Prerequisites

> [!IMPORTANT]
> **Firewall Opening Required**
> 
> All three layers require our server to reach Himkosh's verification API.
> Currently blocked by firewall.
>
> | Parameter | Value |
> |-----------|-------|
> | Source | Our PROD server IP |
> | Destination | `himkosh.hp.nic.in` |
> | Port | **443** (HTTPS) |
> | Protocol | TCP / Outbound |
> | API Endpoint | `https://himkosh.hp.nic.in/eChallan/webpages/AppVerification.aspx` |
> | Purpose | Server-to-server payment verification for automated reconciliation |

---

### Layer 1: Automatic Reconciliation Cron (Background)

**Priority:** HIGH — Eliminates manual SQL fixes entirely

**How it works:**
1. A background cron job runs **every hour**
2. Finds all `himkosh_transactions` where `transaction_status = 'initiated'` AND `created_at < NOW() - INTERVAL '15 minutes'`
3. For each stale transaction, calls the Himkosh **AppVerification.aspx** API with the `app_ref_no`
4. If Himkosh confirms payment success:
   - Updates `transaction_status` → `success`
   - Updates `status` → `Completed successfully.`
   - Stores the GRN in `ech_txn_id`
   - Updates the parent `homestay_applications.status` → `submitted` (if currently `draft`)
   - Logs the reconciliation action in `application_actions` audit table
   - Sends notification to the applicant: "Your payment has been verified"
5. If Himkosh says still pending or failed — logs it, skips to next

**Files to modify/create:**
- `server/himkosh/reconciler.ts` — [NEW] Core reconciliation logic
- `server/himkosh/routes.ts` — Register cron job on server startup
- `server/himkosh/config.ts` — Add verification API config

**Schedule:** Every 60 minutes, with a configurable interval via `system_settings`

---

### Layer 2: Retry on Page Load (User-Triggered, Automatic)

**Priority:** MEDIUM — Provides instant resolution when owner visits dashboard

**How it works:**
1. When an applicant opens their dashboard or payment status page
2. If their latest transaction is `initiated` AND older than 10 minutes
3. Our server calls the Himkosh verification API in real-time
4. If payment confirmed → auto-update and show "Payment Verified ✅"
5. If still pending → show current status with "Verification in progress..."

**User Experience:** The applicant sees their stuck payment resolve **the moment they open the page**, without waiting for the hourly cron.

**Files to modify:**
- `server/himkosh/routes.ts` — Add verification check to payment status endpoint
- `client/src/pages/applications/payment-himkosh-view.tsx` — Show verification status

### Layer 3: Manual Reconciliation Console (Dedicated `accounts` Role)

**Priority:** MEDIUM — Provides officer-level visibility and manual control

**New Role: `accounts`**

A dedicated role for accounts/finance staff who manage payment reconciliation. This role will have a focused, single-purpose dashboard.

**Role Permissions:**

| Feature | Access |
|---------|--------|
| Payment Reconciliation Dashboard | ✅ Full access |
| View all Himkosh transactions | ✅ Read-only |
| Trigger manual verification | ✅ Yes |
| Mark transactions manually | ✅ With audit trail |
| Application data | ❌ No access |
| User management | ❌ No access |
| System settings | ❌ No access |
| DA/DTDO/DO workflows | ❌ No access |

**Dashboard Features:**

#### 3.1 Pending Transactions View
- List of all `initiated` and `redirected` transactions older than 15 minutes
- Shows: Application Number, Owner Name, District, Amount, DDO, Initiated Time, Age
- One-click "Verify with Himkosh" button per transaction
- Bulk "Verify All" button

#### 3.2 Reconciliation History
- Log of all auto and manual reconciliations
- Shows: Who reconciled, when, old status → new status, GRN assigned
- Filterable by date, district, status

#### 3.3 Transaction Search
- Search by Application Number, GRN, CIN, or Owner Name
- Cross-reference our records with Himkosh data
- Side-by-side comparison view (Our System vs Himkosh)

#### 3.4 Summary Dashboard Cards
- Total Pending Reconciliation (count + amount)
- Reconciled Today (count + amount)
- Failed Reconciliation (Himkosh says failed but we show initiated)
- Success Rate (last 30 days)

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Himkosh API changes format | Validate response schema, alert on unexpected format |
| Duplicate reconciliation (cron + page load) | Use DB transaction + idempotent update (only update if still `initiated`) |
| False positive (mark as success when not) | Always verify GRN exists in Himkosh response before marking success |
| Rate limiting by Himkosh | Batch requests with 2-second delay between calls |
| Firewall not opened | Layer 3 manual console still shows pending txns for SQL-based manual fixes |
