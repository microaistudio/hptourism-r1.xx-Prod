# HP Tourism Homestay Portal — Deployment Standard Operating Procedure (SOP)

> **⚠ CRITICAL WARNING: ZOMBIE PROCESS PREVENTION**
> Never run a deployment by simply copying files as `root` and running `pm2 restart`.
> This specifically caused a massive failure where 3,000+ zombie restarts crashed the system due to file ownership conflicts between `root` and the native application user `hptourism`.
> **You must strictly follow the build, chown, and pm2 commands in this document.**

---

## 1. Application Architecture & Constraints

If you ever lose context on how the production environment is structured, here are the absolute constraints:

| Parameter | Value | Critical Note |
| :--- | :--- | :--- |
| **App User** | `hptourism` | All application files and processes must be owned by this user. |
| **Prohibited Users**| `root`, `hmsusr` | Running PM2 as these users WILL spawn zombie crash loops. |
| **Port Mapping** | `5055` | The Node API layer binds to `5055`. Nginx reverse proxies `:443` -> `:5055`. |
| **PM2 Command** | `sudo -u hptourism pm2 <cmd>` | NEVER type just `pm2` as root. Always impersonate `hptourism`. |
| **PM2 Name** | `hptourism-prod` | The registered process name inside the PM2 daemon. |
| **Base Directory** | `/opt/hptourism/homestay/`| Where the live production code runs. |

---

## 2. Generating the Clean Build Package (DEV Environment)

Do not use legacy `build-package.sh` scripts. They blindly zip the entire repository, resulting in 200MB+ bloated files that leak `.env` secrets, developer-centric dummy config files (`ecosystem.config.cjs`), and `node_modules`.

**Always use the secure builder script:**
```bash
cd ~/Projects/hptourism/homestay/
bash scripts/build-prod-pack.sh
```
This script will:
1. Run safety checks to detect leaked hardcoded `dev.osipl.dev` domains.
2. Filter out `.env`, `node_modules`, and `ecosystem.config.cjs`.
3. Output a secure, lightweight (~91MB) `.tar.gz` bundle strictly containing compiled `dist/` and `shared/` folders.

*The package will be deposited into: `~/Projects/setup/Install_packs/hptourism-v<VERSION>-prod.tar.gz`*

## 3. The 6-Step bulletproof PROD Deployment Guide

Once you `scp` your clean tarball up to the PROD server (`root@homstyshmapp:~/setup/`), execute these steps precisely via SSH as `root`.

### Step 1: Backup & Extract
Always maintain an immediate rollback state.
```bash
cd ~/setup
tar -xzf hptourism-v<VERSION>-prod.tar.gz
cp -r /opt/hptourism/homestay/dist /opt/hptourism/homestay/dist_backup_$(date +%s)
```

### Step 2: Deploy New Files
Overwrite the old bundled folders.
```bash
cp -r hptourism-v<VERSION>-prod/dist/* /opt/hptourism/homestay/dist/
cp -r hptourism-v<VERSION>-prod/shared/* /opt/hptourism/homestay/shared/
cp hptourism-v<VERSION>-prod/package.json /opt/hptourism/homestay/package.json
cp -r hptourism-v<VERSION>-prod/migrations/* /opt/hptourism/homestay/migrations/ 2>/dev/null || true
```

### Step 3: THE MOST IMPORTANT STEP (Fix Ownership)
When you copy files as `root` (Step 2), `root` becomes the owner of those fresh files. If you skip this step, the `hptourism` PM2 process gets **Access Denied**, triggering infinite zombie restart loops.
```bash
chown -R hptourism:hptourism /opt/hptourism/homestay/dist/
chown -R hptourism:hptourism /opt/hptourism/homestay/shared/
chown -R hptourism:hptourism /opt/hptourism/homestay/package.json
chown -R hptourism:hptourism /opt/hptourism/homestay/migrations/
```

### Step 4: Run Database Migrations (Drizzle)
Push any new schema changes immediately prior to boot up.
```bash
cd /opt/hptourism/homestay/
sudo -u hptourism bash -c "export \$(grep -v '^#' .env | grep -v '^\s*$' | xargs) && npm run db:push"
```

### Step 5: Safe PM2 Restart
Restart the existing daemon as the dedicated app user and apply environment updates.
```bash
sudo -u hptourism pm2 restart hptourism-prod --update-env
sudo -u hptourism pm2 save
```

### Step 6: Verify Health
Confirm the `.env` did not break and that the port is listening cleanly.
```bash
# PM2 Status Board (Ensure Restarts = 0)
sudo -u hptourism pm2 list

# Live Monitor (Ensure CPU is normal and Heap < 200MB)
sudo -u hptourism pm2 monit

# Internal cURL check
curl -I http://localhost:5055/
```

## 4. Nuclear Option: The Zombie Cleanser Script
If someone accidentally ran `pm2` as root, or permissions got mangled, and the server is locking up with thousands of restarts, follow this routine to fully purge and reset the daemon state:

Create and run a single shell script file on PROD:
```bash
#!/bin/bash
echo "Killing all PM2 processes globally..."
pm2 stop all 2>/dev/null || true; pm2 delete all 2>/dev/null || true
sudo -u hmsusr pm2 stop all 2>/dev/null || true; sudo -u hmsusr pm2 delete all 2>/dev/null || true
sudo -u hptourism pm2 stop all 2>/dev/null || true; sudo -u hptourism pm2 delete all 2>/dev/null || true

echo "Sniping orphaned Node.js threads..."
pgrep -f "node.*dist/index.js" | xargs kill -9 2>/dev/null || true
fuser -k 5055/tcp 2>/dev/null || true

echo "Fixing file ownership forcibly..."
chown -R hptourism:hptourism /opt/hptourism/homestay/

echo "Starting fresh PM2 process cleanly..."
sudo -u hptourism bash -c "cd /opt/hptourism/homestay/ && pm2 start ./dist/index.js --name hptourism-prod --cwd /opt/hptourism/homestay/ --max-memory-restart 1G"
sudo -u hptourism pm2 save
```
