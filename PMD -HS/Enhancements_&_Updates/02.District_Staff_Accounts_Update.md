# District Staff Accounts Update Plan

## Overview

Two enhancements requested by field staff to improve operational flexibility:

1. **Inspector Role** — Allow non-DA personnel (DTDO, Hotel Inspectors, etc.) to conduct field inspections
2. **Multiple DA Accounts** — Enable individual logins per DA staff member for accountability and multi-device access

---

## Part A: Inspector Role Enhancement

### Problem

Currently, only DAs can be assigned inspections. In practice, inspections are also conducted by:
- **DTDO** (District Tourism Development Officer) — already has a login
- **Hotel Inspector** — no login exists
- **Tourism Inspector** — no login exists
- Other district-level inspecting officers

### Solution: New `inspector` Role

Create a single new role `inspector` with inspection-only access. Each inspecting officer gets their own login.

#### Access Matrix

| Feature | `inspector` | `dealing_assistant` | `district_tourism_officer` |
|---------|:-----------:|:-------------------:|:--------------------------:|
| View assigned inspections | ✅ | ✅ | ✅ *(new)* |
| Submit inspection reports | ✅ | ✅ | ✅ *(new)* |
| View application details (read-only) | ✅ | ✅ | ✅ |
| Start scrutiny / verify documents | ❌ | ✅ | ❌ |
| Forward to DTDO | ❌ | ✅ | ❌ |
| Forward to DO / Approve | ❌ | ❌ | ✅ |
| User management | ❌ | ❌ | ❌ |
| Analytics / Reports | ❌ | ❌ | ✅ |
| Grievances | ❌ | ❌ | ✅ |
| System settings | ❌ | ❌ | ❌ |

2. **Backend**:

## Part B: Multiple DA Accounts Per District

### Problem

Currently, one DA login is shared across multiple personnel in a district. This creates two issues:
1. **Multi-device access** — Single-session policy kicks out other machines
2. **No accountability** — Can't audit who actually performed scrutiny/forwarding

### Solution: Individual DA Logins (Recommended)

Create separate accounts for each DA staff member, all assigned to the same district.

#### How It Works

```
District: Chamba
        │
        ├── da_chamba_1 (Asha Sharma)  ──┐
        ├── da_chamba_2 (Ravi Kumar)   ──┤── All see the SAME Chamba applications
        └── da_chamba_3 (Meena Devi)   ──┘   (Shared Queue Model)
```

- All DAs in a district see the **same application queue**
- When DA clicks "Start Scrutiny" → that application is **claimed** by them
- Other DAs see it as "Under Scrutiny" and cannot work on it
- The `daId` field records who handled it → **full audit trail**

#### Shared Queue Model

| Application | Status | Visible to da_chamba_1? | Visible to da_chamba_2? | Owned By |
|------------|--------|:-----------------------:|:-----------------------:|----------|
| CHM-000303 | `submitted` | ✅ Can start scrutiny | ✅ Can start scrutiny | Nobody yet |
| CHM-000288 | `under_scrutiny` | 👁 Read-only | ✅ **Working on it** | da_chamba_2 |
| CHM-000226 | `forwarded_to_dtdo` | 👁 View only | 👁 View only | da_chamba_1 (forwarded) |

#### Benefits

| Benefit | Detail |
|---------|--------|
| **Accountability** | Audit trail shows exactly who did what |
| **Multi-device** | Each person has their own session, no conflicts |
| **Workload visibility** | Analytics can show per-DA productivity |
| **Government compliance** | Individual responsibility for file processing |
| **No code changes** | System already supports multiple DAs per district! |

> [!IMPORTANT]
> **No code changes required for basic functionality.** The existing system already supports multiple DA accounts per district. The DA dashboard uses district-based filtering, not user-ID filtering. Creating additional DA accounts is purely an admin operation.

#### Optional Enhancements (Future)

These are nice-to-have improvements if multiple DAs become common:

| Enhancement | Description |
|-------------|-------------|
| **"Claimed by" indicator** | Show which DA is working on an application |
| **Per-DA stats** | Analytics breakdown showing applications processed per DA |
| **Auto-assign** | Round-robin assignment of new applications to available DAs |
| **Workload balancing** | Alert when one DA has too many pending cases |

#### Setup Steps

1. **Admin Action Only**: Go to District Staff → Create User
2. Enter individual details (name, mobile, email)
3. Set role = `dealing_assistant`
4. Set district = same district as existing DA
5. Share credentials with the person
6. Done — they can immediately log in and see the district queue
